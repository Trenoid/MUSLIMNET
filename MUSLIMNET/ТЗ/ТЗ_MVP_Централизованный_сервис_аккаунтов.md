# Техническое задание (ТЗ) на MVP
## Централизованный сервис аккаунтов MUSLIMNET

## 1. Общие положения

### 1.1 Цель документа
Зафиксировать формальные требования к реализации MVP централизованного сервиса аккаунтов, аутентификации, авторизации, ограничений, аудита и администрирования для экосистемы MUSLIMNET.

### 1.2 Основание
ТЗ сформировано на базе концептуальных документов каталога `MUSLIMNET` (введение, модель доступа, роли и ограничения, жизненный цикл аккаунта, безопасность, продуктовые примеры, открытые вопросы).

### 1.3 Границы MVP
В первую очередь должны быть покрыты:
- единый аккаунт для нескольких приложений;
- централизованный вход и проверка сессий;
- двухуровневая авторизация (доступ к продукту + действие);
- роли, точечные права и ограничения;
- административная панель базового уровня;
- аудит критических операций.

Вне MVP (следующие итерации):
- расширенная аналитика рисков;
- сложные автоматические политики комплаенса;
- полная многоорганизационная модель с кастомными юридическими регламентами.

## 2. Функциональные требования

### 2.1 Подсистема идентификации и аутентификации
Система должна обеспечивать:
1. Регистрацию учетной записи по утвержденному сценарию (открытая/инвайт-модель определяется конфигурацией окружения).
2. Вход по логину (email или username) и паролю.
3. Выход из текущей сессии.
4. Выход из всех сессий пользователя.
5. Обновление access-токена через refresh-токен.
6. Сброс пароля через подтвержденный канал.
7. Поддержку MFA (минимум TOTP) как опциональную возможность в MVP и обязательную — для привилегированных ролей.

### 2.2 Подсистема управления аккаунтом
Система должна поддерживать:
1. Профиль пользователя (базовые поля, контакты, статус верификации).
2. Статусы аккаунта: `pending`, `active`, `restricted`, `blocked`, `deleted`.
3. Переходы между статусами по регламенту (см. раздел 4).
4. Историю изменений статуса и оснований.
5. Привязку/отвязку контактных данных.

### 2.3 Подсистема авторизации и прав доступа
Система должна реализовывать модель:
- **L1: Product Access** — доступ к приложению в целом.
- **L2: Action Access** — доступ к конкретным действиям.

Требования:
1. Проверка допуска пользователя к каждому приложению.
2. Проверка конкретного permission перед выполнением действия.
3. Поддержка ролей уровня платформы и ролей уровня продукта.
4. Поддержка выдачи индивидуальных permission напрямую пользователю.
5. Поддержка приоритета deny-правил над allow.
6. Возможность временного назначения прав (start/end).

### 2.4 Подсистема ограничений и санкций
Система должна поддерживать:
1. Ограничения уровня продукта (например: запрет публикации, запрет комментариев).
2. Ограничения уровня действия (например: запрет загрузки файлов).
3. Временные и бессрочные ограничения.
4. Хранение основания, автора и срока санкции.
5. Просмотр активных ограничений при каждом запросе авторизации.
6. Ручное снятие санкции уполномоченным администратором.

### 2.5 Подсистема администрирования
Должна быть реализована административная панель (минимальный набор):
1. Поиск пользователя по ID/email/username.
2. Просмотр карточки пользователя (статус, роли, права, ограничения, сессии).
3. Назначение и снятие ролей.
4. Выдача и отзыв точечных permissions.
5. Установка и снятие ограничений.
6. Принудительное завершение сессий.
7. Изменение статуса аккаунта с обязательным обоснованием.
8. Просмотр аудита действий по пользователю.

### 2.6 Подсистема аудита
Система должна журналировать:
1. Входы/выходы, сбросы паролей, MFA-события.
2. Изменение ролей, прав, статусов, ограничений.
3. Административные операции высокой критичности.
4. Неуспешные попытки доступа и авторизационные отказы.

Для каждой записи аудита обязательно:
- `event_id`, `timestamp`, `actor_id`, `subject_id`, `action`, `result`, `ip`, `user_agent`, `correlation_id`.

## 3. Нефункциональные требования

### 3.1 Производительность
1. 95-й перцентиль ответа API авторизации: не более 200 мс при номинальной нагрузке.
2. 99-й перцентиль: не более 500 мс.
3. Проверка permission в кэше: не более 20 мс.

### 3.2 Доступность
1. Целевой SLA MVP: 99.9%.
2. Деградационный режим для интегрируемых приложений: при недоступности центра приложение должно корректно переходить в read-only/ограниченный режим по заранее заданной стратегии.

### 3.3 Безопасность
1. Хранение паролей только в виде стойкого хэша (Argon2id или эквивалент).
2. Токены подписываются ротационными ключами.
3. Защита от brute-force (rate limit + временная блокировка).
4. TLS для всех межсервисных и клиентских соединений.
5. Обязательный аудит для критических операций.

### 3.4 Наблюдаемость
1. Метрики по аутентификации, отказам авторизации, ошибкам API, задержкам.
2. Структурированные логи и трассировка с `correlation_id`.
3. Алерты по аномальному росту отказов и подозрительных входов.

## 4. Жизненный цикл аккаунта

### 4.1 Статусы
- `pending` — учетная запись создана, но не активирована/не подтверждена.
- `active` — рабочий доступ разрешен.
- `restricted` — доступ ограничен частично.
- `blocked` — доступ полностью запрещен.
- `deleted` — аккаунт деактивирован согласно политике удаления.

### 4.2 Допустимые переходы
1. `pending -> active`
2. `active -> restricted`
3. `restricted -> active`
4. `active -> blocked`
5. `restricted -> blocked`
6. `blocked -> active` (только для ролей повышенных полномочий)
7. `active|restricted|blocked -> deleted` (по регламенту жизненного цикла данных)

Каждый переход фиксируется в аудите с причиной и автором.

## 5. Модель данных (MVP)

Минимальные сущности:
1. `users`
2. `credentials`
3. `sessions`
4. `applications`
5. `roles`
6. `permissions`
7. `role_permissions`
8. `user_roles`
9. `user_permissions`
10. `restrictions`
11. `audit_events`

Минимальные ключевые связи:
- пользователь ↔ роли (многие-ко-многим);
- роль ↔ права (многие-ко-многим);
- пользователь ↔ точечные права (многие-ко-многим);
- пользователь ↔ ограничения (один-ко-многим);
- пользователь ↔ сессии (один-ко-многим).

## 6. API-контракты (минимум для MVP)

### 6.1 Внешние API (для клиентов и приложений)
1. `POST /auth/register`
2. `POST /auth/login`
3. `POST /auth/logout`
4. `POST /auth/refresh`
5. `POST /auth/password/reset/request`
6. `POST /auth/password/reset/confirm`
7. `GET /me`
8. `GET /me/permissions?app={app_id}`
9. `POST /auth/check-access` (проверка допуска к приложению/действию)

### 6.2 Административные API
1. `GET /admin/users/{id}`
2. `POST /admin/users/{id}/roles`
3. `DELETE /admin/users/{id}/roles/{role_id}`
4. `POST /admin/users/{id}/permissions`
5. `DELETE /admin/users/{id}/permissions/{permission_id}`
6. `POST /admin/users/{id}/restrictions`
7. `DELETE /admin/users/{id}/restrictions/{restriction_id}`
8. `POST /admin/users/{id}/status`
9. `POST /admin/users/{id}/sessions/revoke`
10. `GET /admin/audit?subject_id=&actor_id=&from=&to=`

### 6.3 Внутренние требования к API
- Версионирование `/v1`.
- Идемпотентность для операций, где это возможно.
- Единый формат ошибок (`code`, `message`, `details`, `correlation_id`).

## 7. Интеграция с продуктами экосистемы

MVP интеграции: видеохостинг, мессенджер, файловое хранилище.

Требования к интеграции каждого продукта:
1. Перенаправление на единый вход.
2. Проверка токена и сессии через центральный сервис.
3. Проверка `product_access` при входе в продукт.
4. Проверка `action_permission` перед критичными действиями.
5. Корректная обработка ограничений и кодов отказа.
6. Передача `correlation_id` в межсервисных вызовах.

---

## 8. Ролевая модель операторов

Минимальные административные роли MVP:
1. `super_admin` — полный контроль, включая разблокировку и управление ролями админов.
2. `security_admin` — безопасность, сессии, MFA-политики, расследование инцидентов.
3. `support_admin` — ограниченные операции поддержки (без критичных изменений ролей верхнего уровня).
4. `product_admin` — управление доступом в рамках конкретного продукта.
5. `moderator` — санкции контентного уровня в рамках продукта.

Критичные операции (смена статуса на `blocked`, выдача `super_admin`, массовый отзыв сессий) должны требовать усиленный контроль (например, подтверждение вторым администратором — опционально для MVP, обязательно в следующей фазе).

## 9. План реализации MVP (этапы)

### Этап 1. База идентификации
- модели пользователей, учетных данных и сессий;
- регистрация, вход, выход, refresh;
- базовый профиль и статус аккаунта.

### Этап 2. Авторизация
- модели ролей и permissions;
- проверка product/action доступа;
- API `check-access`.

### Этап 3. Ограничения и админ-панель
- ограничения (restrictions);
- админ-операции по ролям/правам/статусам/сессиям;
- базовые сценарии поддержки.

### Этап 4. Аудит и безопасность
- полноценный аудит событий;
- MFA для привилегированных ролей;
- rate limiting и защита от brute-force.

### Этап 5. Интеграция приложений первой очереди
- видеохостинг;
- мессенджер;
- файловое хранилище.

---

## 10. Критерии приемки MVP

MVP считается принятым, если выполнены условия:
1. Пользователь может зарегистрироваться, войти и использовать единый аккаунт минимум в 3 продуктах.
2. Продуктовый доступ и доступ к действиям проверяются централизованно.
3. Роли и точечные permissions применяются корректно, включая deny-приоритет.
4. Ограничения применяются и снимаются через админ-интерфейс и отражаются в проверках доступа.
5. Все критичные админ-действия присутствуют в аудите.
6. Базовые нефункциональные показатели (раздел 3) подтверждены тестами.

## 11. Риски и зависимости

### 11.1 Риски
1. Рассинхронизация прав между сервисами при некорректном кэшировании.
2. Сложность миграции существующих аккаунтов.
3. Недостаточная формализация административных регламентов.

### 11.2 Зависимости
1. Единая инфраструктура секретов и ключей подписи.
2. Централизованный мониторинг и логирование.
3. Наличие команд интеграции со стороны продуктовых сервисов.
